Architecture Guide

Gym Platform Project Walkthrough

Complete project explanation for the Spring Boot backend in D:\Codex-Gym. This guide explains the purpose of each class, how the layers integrate, and how the important business flows move through the system.

Runtime Shape Spring Boot API + PostgreSQL + RabbitMQ + optional Email / Telegram / WhatsApp channels.
Main Business Areas Members, membership plans, memberships, invoices, payments, reminders, receipts, and cards.
Design Style Controller -> application service -> repository / helper service -> integration gateway.
Java 21 / 22 runtime compatible Spring Boot 3.3 JPA + PostgreSQL RabbitMQ notifications Scheduler driven reminders Telegram connect flow

1. Project In One Page

This project is a gym operations backend. It stores member and billing data in PostgreSQL, exposes REST APIs for frontends or operators, and sends notifications asynchronously through RabbitMQ. The code is intentionally layered so that HTTP handling, business rules, persistence, document generation, and external delivery are separate concerns.

Client / Frontend -> REST Controllers -> Application Services -> Repositories -> PostgreSQL -> Helper Services (mapping, code generation, document generation, message text generation) -> Application Events -> MemberCardAutomationListener -> NotificationPublisher -> RabbitMQ exchange -> queue -> NotificationConsumer -> EmailGateway / TelegramGateway / WhatsAppGateway Scheduled jobs -> FeeReminderScheduler -> NotificationService -> same RabbitMQ path as manual notifications
The notification system is asynchronous by design. The database decides what should be sent, NotificationService records a PENDING log, RabbitMQ transports the work, and NotificationConsumer performs the actual channel delivery.

2. Boot And Runtime Flow

Application start -> GymPlatformApplication -> enables scheduling -> enables JPA auditing -> TimeConfiguration creates Clock bean -> WebCorsConfig loads CORS settings -> RabbitMqNotificationConfig creates exchange / queue / DLQ / RabbitTemplate -> Spring wires repositories, services, controllers, listeners, scheduler -> DevelopmentDataInitializer seeds membership plans if database is empty -> API becomes available on port 8080

Central runtime configuration comes from src/main/resources/application.yml and container overrides via environment variables.

3. End-To-End Business Flows

3.1 Member creation and automatic member card

POST /api/v1/members -> MemberController.createMember -> MemberService.createMember -> generate member code -> validate / normalize input -> decode profile image -> MemberRepository.save -> publish MemberCreatedEvent -> transaction commits -> MemberCardAutomationListener.onMemberCreated -> NotificationService.queueMemberCard -> MemberCardService.prepareMemberCard -> MemberCardDocumentFactory.buildMemberCard -> create NotificationLog(PENDING) -> publish NotificationEventMessage to RabbitMQ -> NotificationConsumer -> EMAIL or TELEGRAM channel gateway -> mark log SENT or FAILED

3.2 Membership creation and invoice generation

POST /api/v1/memberships -> MembershipController.createMembership -> MembershipService.createMembership -> load member -> load plan -> validate active member + active plan -> create Membership -> BillingService.createInvoice -> build membership signup invoice -> publish MembershipCardRefreshEvent -> return MembershipEnrollmentResponse -> after commit listener can queue updated membership card notification

3.3 Invoice payment and receipt notification

POST /api/v1/invoices/{invoiceId}/payments -> InvoiceController.recordPayment -> BillingService.recordPayment -> validate payment amount -> save Payment -> update Invoice amountPaid / balanceDue / status Optional next step POST /api/v1/invoices/{invoiceId}/receipt-notifications -> NotificationService.queueInvoiceReceipt -> build receipt attachment with InvoiceDocumentFactory -> queue member and admin notifications -> publish through RabbitMQ Alternative next step POST /api/v1/invoices/{invoiceId}/receipt-notifications/pdf -> upload an external PDF -> NotificationService.queueInvoiceReceiptPdf -> send uploaded PDF instead of generated HTML receipt

3.4 Daily reminders

Scheduled cron (default 9:00 AM) -> FeeReminderScheduler.pushScheduledReminders -> MembershipService.refreshExpiredMemberships -> BillingService.refreshOverdueInvoices -> find due / overdue invoices -> find active memberships ending soon -> NotificationService.was...SentToday checks NotificationLogRepository -> queueInvoiceReminder / queueRenewalReminder -> publish to RabbitMQ -> NotificationConsumer delivers channel messages

3.5 Telegram connect flow

POST /api/v1/members/{memberId}/telegram-connect -> TelegramConnectionService.createConnectSession -> ensure Telegram is enabled and not dry-run -> create TelegramConnectionToken -> build deep link: https://t.me/{bot}?start=connect_{token} Later sync step POST /api/v1/members/telegram-connections/sync -> TelegramConnectionService.syncConnections -> TelegramGateway.getUpdates -> parse /start connect payload -> find token -> ensure not expired / not consumed / no chat conflict -> save member.telegramChatId -> mark token consumed -> return sync report

14. Integration Notes And Key Design Decisions

  • Why RabbitMQ exists: the app does not send external messages directly inside core request paths. It queues work so APIs stay faster and external outages are isolated.
  • Why notification logs exist: they provide an audit trail and allow "was this reminder already sent today?" checks.
  • Why a shared Clock bean exists: business logic and tests can agree on time without calling LocalDate.now() everywhere.
  • Why document factories return HTML attachments: the system currently generates styled HTML files for receipts and cards. A separate uploaded-PDF flow exists for invoice receipts when a real PDF file is available.
  • Why application events are used: member-card delivery is triggered after a successful transaction commit, which avoids sending cards for rolled-back writes.
  • Why WhatsApp uses delegation: environments without a real webhook still work through a dry-run logging adapter.
  • Why Telegram link tokens exist: they prevent arbitrary chat-id assignment and create a safer "deep-link then confirm" integration flow.

16. Final Mental Model

If you remember only one mental picture, make it this: the project is a CRUD-and-workflow backend where memberships create invoices, payments update invoices, notifications are queued asynchronously, and cards / receipts are generated as attachments. Controllers stay thin, application services own rules, repositories own persistence, and RabbitMQ separates business decisions from external message delivery.
